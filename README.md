# Art Jewelry

A single-page site for **artjewelry.work** with three contact actions (Email,
Phone, WhatsApp), private visitor analytics, and an encrypted `/admin`
dashboard — the same login + analytics design as nervemedia.agency, on a
simple dark background.

```
index.html            Front-end (Email / Call / WhatsApp buttons)
style.css             Front-end styles (mobile-first)
script.js             Small front-end helper
analytics.js          Anonymous visit + click tracking (Firebase)
firebase-config.js    PUBLIC Firebase web config  ← you paste your values here
firebase.json         Points Firebase at the database rules
database.rules.json   Realtime Database security rules
admin/
  index.html          Admin login + dashboard shell
  admin.css           Self-contained admin styles (simple dark background)
  admin.js            Login flow: AES-GCM decrypt + Firebase Auth (both must pass)
  stats.js            Reads /users from Firebase and renders the dashboard
  stats-core.mjs      Pure stats math (testable, no Firebase)
  payload.js          Encrypted blob, generated by tools/encrypt-admin.mjs
tools/
  encrypt-admin.mjs   Regenerates admin/payload.js from the admin password
```

## Firebase setup (do this once)

You build the project; everything in this repo is already wired to it.

### 1. Create the project
1. Go to <https://console.firebase.google.com> → **Add project** (e.g. `artjewelry`).
2. **Build → Authentication → Get started**, then enable two providers:
   - **Anonymous** (lets visitors be counted)
   - **Email/Password**
3. In **Authentication → Users → Add user**, create the admin login:
   - Email: `admin@artjewelry.work`
   - Password: `***REMOVED***`

   > The email is just the Firebase Auth identifier — it does **not** need to be
   > a real mailbox, but it must match exactly (it's hard-coded in
   > `admin/stats.js` and in `database.rules.json`).

### 2. Create the Realtime Database
1. **Build → Realtime Database → Create database**.
2. Pick a region (e.g. `europe-west1`) and start in **locked mode**.
3. Go to the **Rules** tab, paste the full contents of `database.rules.json`
   from this repo, and **Publish**.

   (Or, with the Firebase CLI: `firebase deploy --only database`.)

### 3. Paste the web config
1. **Project settings (gear) → General → Your apps → Web app** (`</>`). Register
   an app if you haven't.
2. Copy the `firebaseConfig` values into **`firebase-config.js`**, replacing the
   `PASTE_…` placeholders. Make sure `databaseURL` is included — it looks like
   `https://<project>-default-rtdb.<region>.firebasedatabase.app`.

   These values are **public** identifiers (safe to commit / serve). Real
   security comes from the database rules above.

### 4. Authorize your domain
In **Authentication → Settings → Authorized domains**, add `artjewelry.work`
(and `www.artjewelry.work` if you use it). `localhost` is there by default for
local testing.

That's it — visit the site, then open `/admin/` and sign in.

## Changing the admin password

The password is used in two places, so update both:

1. **Firebase:** Authentication → Users → reset the password for
   `admin@artjewelry.work`.
2. **Encrypted payload:** regenerate `admin/payload.js` with the new password:

   ```powershell
   # PowerShell
   $env:ADMIN_PASSWORD = "your-new-password"; node tools/encrypt-admin.mjs
   ```
   ```bash
   # bash
   ADMIN_PASSWORD="your-new-password" node tools/encrypt-admin.mjs
   ```

   Commit the regenerated `admin/payload.js`.

## Hosting notes (GitHub Pages + custom domain + Cloudflare)

- This is a static site — GitHub Pages serves it as-is. No build step.
- The admin page sets a strict Content-Security-Policy that already allows the
  Firebase endpoints; nothing to change for the custom domain.
- When you move DNS to Cloudflare later, leave proxying on; the site is static
  and Firebase is called directly from the browser, so no extra config is
  needed. Just keep the domain listed under Firebase **Authorized domains**.
